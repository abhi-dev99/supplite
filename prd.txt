SECTION 1: EXECUTIVE SUMMARY
--------------------------------------------------------------------------------

This system solves WSI's peak-season inventory blindspot: replenishment decisions
must be made 8-12 weeks before demand arrives, but current forecasting tools are
backward-looking and cannot detect demand signals until it is too late to act.

The solution fuses three signal types — historical sales, real-time search virality
(Google Trends), and real estate activity (US Census building permits) — into a
unified demand forecast model that flags SKU-level risk 4–12 weeks in advance.
A large language model then converts model output into plain-language buyer briefs
with specific action recommendations and dollar-impact framing.

Key differentiator: No other system in this hackathon uses housing permit data
as a demand signal. The WSI PS document itself names this signal in Scenario 4.
This is the innovation the judges wrote into the problem statement.

Core value proposition:
"Every other forecasting system sees demand after it happens.
 This one sees it 60 days before — and tells buyers exactly what to do about it."

SECTION 2: PROBLEM STATEMENT
--------------------------------------------------------------------------------

2.1 The Core Problem
  WSI places furniture orders with 8-12 week lead times from overseas vendors.
  Demand can spike suddenly (viral social trends) or gradually (housing market activity).
  There is no system that detects these demand signals early enough for buyers to act
  within the lead time window.

2.2 Simultaneous Failure Modes
  Failure Mode A — Stockout: A product goes viral, demand 3x's in 2 weeks.
    The buyer's last order was placed based on historical averages. Stockout arrives
    in 4 weeks. Reorder lead time is 10 weeks. Nothing can be done.

  Failure Mode B — Overstock: A trend peaks undetected. Slow search decline signals
    demand has dropped, but no system alerts the buyer. 4,000+ units pile up, requiring
    markdown and consuming warehouse space and working capital.

  Both failures occur simultaneously across different SKUs in the same company
  because there is no unified view and no forward-looking signal layer.

2.3 Root Causes
  1. Demand forecasting uses only historical sales — a lagging indicator
  2. Social virality signals are unstructured and not integrated into any model
  3. Real estate activity (a 60-day leading indicator for furniture demand) is
     not tracked by any retail forecasting system
  4. Buyers receive raw model output (numbers) rather than plain-language guidance
  5. Overstock and stockout risk are tracked in separate systems with no unified view

2.4 Scope of Impact
  - WSI peak season (Oct-Dec) represents approximately 35% of annual revenue (~$6.8B/yr)
  - Out-of-market emergency shipments are a documented margin erosion factor (earnings calls)
  - A 10-week lead time means October decisions determine December outcomes

SECTION 5: FUNCTIONAL REQUIREMENTS
--------------------------------------------------------------------------------

5.1 Data Ingestion

FR-D1: The system SHALL ingest historical sales data from sales.csv
  Fields: sku_id, date, units_sold, store_id, category, price
  Format: CSV, pandas DataFrame
  Frequency: Load once at startup; refresh on demand via UI button

FR-D2: The system SHALL fetch Google Trends search index for product keywords
  Tool: pytrends Python library
  Granularity: Weekly (52 data points per year)
  Geography: US national; optionally state-level
  Keywords: Derived from SKU category and product name
    - Bedding: ["pottery barn throw blanket", "cozy throw blanket", "plaid blanket"]
    - Furniture: ["pottery barn sofa", "west elm sofa", "couch"]
    - Decor: ["decorative pillow", "throw pillow set"]
  Rate limiting: 1 second sleep between requests; cache to SQLite on first fetch
  Fallback: If pytrends fails, use pre-cached CSV of trends data

FR-D3: The system SHALL ingest US Census Building Permits data
  Source: census.gov/construction/bps — download monthly CSV
  Fallback: census.gov API with free API key
  Fields: month, metro_area, new_single_family_permits, new_multi_family_permits,
          renovation_permits
  Geographic mapping: Census metro areas → WSI DC service regions
    (Phoenix metro → AZ DC; Charlotte metro → NC DC; etc.)
  Update frequency: Monthly

FR-D4: The system SHALL ingest FRED mortgage application data
  Tool: fredapi Python library
  Series: MBAVCH (MBA Purchase Applications Index), MORTGAGE30US
  API key: Required (free FRED registration)
  Fallback: Pre-downloaded CSV of historical data
  Update frequency: Weekly

FR-D5: The system SHALL integrate all signals into a unified signals DataFrame
  Schema: date | sku_id | signal_type | value | source
  Signal types: 'sales_history' | 'google_trends' | 'housing_permit' | 'mortgage_apps'
  This extends the PS-provided signals.csv schema — housing_permit is a new signal_type

FR-D6: The system SHALL compute inventory metrics from inventory.csv
  Key derived fields:
    - rolling_7day_avg_sales = mean(units_sold, last 7 days)
    - days_of_supply = stock_on_hand / rolling_7day_avg_sales
    - stockout_days_until = days_of_supply (if < lead_time_days, stockout risk is active)
    - inventory_coverage_ratio = days_of_supply / lead_time_days

5.2 Feature Engineering

FR-F1: The system SHALL compute lag features for all signals
  Lags: 1 week, 2 weeks, 4 weeks, 6 weeks, 8 weeks, 12 weeks
  For each signal type: [signal]_lag_1w, [signal]_lag_4w, [signal]_lag_8w, etc.
  Purpose: Model learns which lag distance is most predictive per signal

FR-F2: The system SHALL compute velocity features (rate of change)
  search_velocity_1w = (search_index_now - search_index_1w_ago) / search_index_1w_ago
  permit_velocity_4w = (permits_now - permits_4w_ago) / permits_4w_ago
  sales_velocity_2w = (avg_sales_this_2w - avg_sales_prior_2w) / avg_sales_prior_2w
  These are the KEY features for anomaly detection

FR-F3: The system SHALL compute seasonal features
  is_peak_season: Boolean flag (True for weeks 44-52, Oct-Dec)
  week_of_year: Integer 1-52
  month: Integer 1-12
  quarter: Integer 1-4

FR-F4: The system SHALL compute inventory risk pre-flags
  stockout_risk_pre: 1 if days_of_supply < lead_time_days * 1.2, else 0
  overstock_risk_pre: 1 if days_of_supply > lead_time_days * 3, else 0

FR-F5: The system SHALL encode categorical features
  brand: Label-encoded (Pottery Barn=0, West Elm=1, Williams Sonoma=2, etc.)
  category: Label-encoded (Bedding=0, Furniture=1, Decor=2, etc.)

5.3 Machine Learning Models

FR-M1: Demand Forecast Model (LightGBM Regressor)
  Target variable: units_sold, 30/60/90 days forward
  Train/test split: TimeSeriesSplit with n_splits=5 (no data leakage)
  Features: All lag features + velocity features + seasonal features + category + brand
  Hyperparameters (reasonable defaults for 56-hour build):
    n_estimators=500, learning_rate=0.05, num_leaves=31, min_child_samples=20
  Output: Point forecast per SKU per time horizon
  Model persistence: Save with joblib to /models/demand_forecast.pkl

FR-M2: Anomaly Detector (Isolation Forest + statistical z-score)
  Purpose: Detect sudden demand velocity changes in EITHER direction (spike or collapse)
  Isolation Forest configuration:
    contamination=0.05 (expect ~5% anomalous observations)
    n_estimators=100, max_samples='auto', random_state=42
  Input features: search_velocity_1w, search_velocity_2w, sales_velocity_2w,
                  permit_velocity_4w (when available)
  Z-score fallback: flag if |z_score| > 2.5 on any velocity feature
  Output: anomaly_score (-1 = anomaly, 1 = normal), anomaly_flag (Boolean)
  Trigger condition: anomaly_flag=True AND stockout_risk_pre=1 → STOCKOUT_RISK
                     anomaly_flag=True AND overstock_risk_pre=1 → OVERSTOCK_RISK

FR-M3: Risk Classifier (Rule-based, deterministic)
  Output: 'STOCKOUT_RISK' | 'OVERSTOCK_RISK' | 'OK' | 'WATCH'
  Rules (apply in order):
    STOCKOUT_RISK: days_of_supply < lead_time_days AND forecast_demand > stock_on_hand + on_order
    OVERSTOCK_RISK: days_of_supply > lead_time_days * 3 AND sales_velocity_2w < -0.15
    WATCH: anomaly_flag=True AND NOT (STOCKOUT_RISK or OVERSTOCK_RISK)
    OK: all other cases

5.4 LLM Buyer Brief Generator

FR-L1: The system SHALL generate a weekly plain-language buyer brief via LLM API
  Supported LLMs: Claude API (primary), OpenAI API (fallback), Ollama/Llama 3 (offline fallback)
  API client: anthropic Python library (pip install anthropic)
  Model: claude-sonnet-4-6 (or claude-haiku-4-5-20251001 for cost/speed)

FR-L2: Input to LLM SHALL be a structured JSON context object
  Required fields:
    brief_date: str (YYYY-MM-DD)
    urgent_skus: list of {
      sku_id, product_name, brand, category,
      current_stock: int, on_order: int, lead_time_days: int,
      days_of_supply: float, risk_level: str,
      forecast_demand_60d: int, demand_shortfall: int (if stockout),
      primary_signal: str (which signal is driving this),
      signal_detail: str (e.g., "search volume +840% WoW", "permits +34% MoM Phoenix")
    }
    overstock_skus: list of same structure
    watch_skus: list of same structure
    week_summary: str (brief context on market conditions this week)

FR-L3: The LLM prompt SHALL follow this structure (prompt engineering spec):
  System prompt:
    "You are a demand intelligence analyst for Williams-Sonoma, Inc. You write concise,
    actionable weekly inventory briefs for supply chain buyers. Every recommendation
    must be specific (name the SKU, name the action, name the dollar impact where possible).
    Buyers are experienced professionals — no preamble, no generic advice.
    Format: URGENT ACTION → OVERSTOCK WARNING → WATCH LIST.
    Keep the entire brief under 400 words. Use clear section headers."

  User prompt:
    "Generate this week's demand intelligence brief based on the following data:
    [JSON context object]

    For each urgent SKU: state the risk, cite the signal that caught it, give a
    specific recommended action (e.g., 'expedite supplementary order for X units',
    'consider markdown trigger at Y% discount', 'monitor for 2 more weeks before acting').
    Where possible, estimate the financial stakes (e.g., 'shortfall of 12,200 units
    at $[price] = $[revenue at risk]')."

FR-L4: The system SHALL handle LLM API errors gracefully
  Retry: 3 attempts with exponential backoff (2s, 4s, 8s)
  Fallback: If all retries fail, display pre-templated brief with model outputs injected
  Cache: Save generated briefs to SQLite with (date, model_version) as key
  Rate limiting: Implement token-aware throttling

FR-L5: The brief output SHALL be displayed in the dashboard with streaming
  Use: st.write_stream() or character-by-character display simulation
  Goal: Judges see the AI generating the brief in real time — high visual impact
