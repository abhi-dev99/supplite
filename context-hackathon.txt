PART 6: CORE SYSTEM ARCHITECTURE
--------------------------------------------------------------------------------

System name: Suggested names — "SignalStock", "PulseChain", "FuseFC" (team to decide)

The system has 5 layers:

LAYER 1: Data Ingestion (all sources free, all legal)
  Source A — Sales history: provided sales.csv (synthetic/real WSI data)
  Source B — Google Trends via pytrends: search volume for product keywords
              e.g., "pottery barn throw blanket", "west elm sofa", "pottery barn pillow"
              This is the VIRALITY/SOCIAL PROXY — no Instagram, no scraping
              Weekly granularity, DMA-level geography, free, no API key
  Source C — US Census Building Permits Survey (census.gov/construction/bps)
              Monthly permit data by county/metro area
              This is the REAL ESTATE INTEGRATION — 60-90 day leading indicator
              Free CSV download, no API key required
  Source D — FRED Economic Data (fred.stlouisfed.org)
              MBA Purchase Applications Index, 30-year mortgage rate, housing starts
              Free API (free FRED API key required)
              Python: fredapi (pip install fredapi)
  Source E — Weather/events: OpenWeatherMap free tier or static holiday calendar
              Seasonal and event-driven demand adjustment

LAYER 2: Feature Engineering
  Per SKU × week time series:
  - units_sold_lag_1w, _4w, _8w, _12w (historical sales with lags)
  - search_volume (Google Trends index for SKU keywords)
  - search_velocity (rate of change in search volume — key for viral detection)
  - permit_count_lag_4w, _8w, _12w (building permits 30/60/90 days ago)
  - mortgage_apps_lag_6w (FRED mortgage applications 6 weeks prior)
  - is_peak_season (Thanksgiving-Christmas window flag)
  - holiday_flag, weather_deviation
  - stock_on_hand, on_order, lead_time_days (from inventory.csv)
  - days_of_supply = stock_on_hand / rolling_7day_avg_sales
  - overstock_flag: days_of_supply > 90
  - stockout_risk_flag: days_of_supply < lead_time_days
  - brand (categorical: Pottery Barn, West Elm, Williams Sonoma, etc.)
  - category (categorical: Bedding, Furniture, Decor, etc.)

LAYER 3: ML Models
  Model A — Demand Forecast: LightGBM Regressor
    - Target: units_sold 30/60/90 days forward per SKU
    - Input: all engineered features above
    - Training: TimeSeriesSplit (scikit-learn) — NO data leakage
    - Output: point forecast + prediction interval

  Model B — Anomaly Detector: Isolation Forest (sklearn) + statistical z-score
    - Input: demand velocity features (rate of change in search + sales)
    - Output: anomaly score per SKU per week
    - Flags when demand velocity suddenly changes (either direction)
    - Specifically calibrated to catch the "viral spike" scenario

  Model C — Risk Classifier: Rule-based (deterministic)
    - Input: days_of_supply, lead_time_days, anomaly score, forecast
    - Output: STOCKOUT_RISK / OVERSTOCK_RISK / OK per SKU
    - Simple thresholds: days_of_supply < lead_time_days → STOCKOUT_RISK
    - days_of_supply > 3 × avg_selling_days + lead_time → OVERSTOCK_RISK

LAYER 4: LLM Buyer Brief Generator (KEY DEMO MOMENT)
  Framework: Call Claude/GPT API (free tier or Anthropic API)
  Input: structured JSON with {sku_id, forecast, current_inventory, signals_driving_change,
         risk_level, lead_time_days, recommended_action}
  Prompt: structured prompt engineering to generate a weekly buyer brief
  Output format:
    "WEEKLY DEMAND INTELLIGENCE BRIEF — Week of [date]

    URGENT — ACTION REQUIRED (3 SKUs)
    1. PB-BLANKET-42 (Pottery Barn Throw Blanket, Cognac)
       Current stock: 6,200 units | Lead time: 10 weeks | Days of supply: 34 days
       SIGNAL: Search volume up 840% in 7 days (social virality detected)
       FORECAST: 18,400 units demand by Dec 15 — 12,200 unit SHORTFALL
       ACTION: Expedite supplementary order from alternate vendor. Consider air freight
               for 3,000 units at $X premium vs $Y in lost sales.

    OVERSTOCK WARNING (2 SKUs)
    2. PB-PILLOW-71 (Decorative Pillow, Sage)
       Stock: 4,200 units | Selling rate: 47/week | Days of supply: 89 days
       SIGNAL: Search declining 23% over 8 weeks | Trend peaked February
       ACTION: Consider promotion or markdown trigger. Avoid reorder."

  This is the "wow" demo moment — judges see AI generating a real, actionable business document.

LAYER 5: Dashboard (Streamlit)
  Panel 1: SKU Risk Overview
    - Table of all SKUs with current stock, days of supply, risk status (color-coded)
    - Filters: brand, category, risk level
    - Click any SKU to drill down

  Panel 2: Demand Signal Timeline (per SKU, click from Panel 1)
    - Multi-line chart: historical sales + 90-day forecast + confidence interval
    - Overlay: search trend index (Google Trends) on secondary axis
    - Overlay: permit activity (housing permits) on tertiary axis
    - Vertical line: "signal detected" moment
    - Shows signal → forecast → inventory gap visually

  Panel 3: Weekly LLM Buyer Brief
    - Button: "Generate This Week's Brief"
    - Calls LLM API, streams output into the dashboard
    - Download as PDF option
    - Shows "AI reasoning" expandable section (which signals triggered each recommendation)

  Panel 4: Simulation Mode ("What Would Have Happened")
    - User selects a past date (e.g., March 12, 2026)
    - System shows: what signals were present, what the model would have flagged,
      vs what actually happened (stockout/overstock)
    - This directly addresses the PS's suggestion of "a simulation showing what would have happened"

PART 7: DATA SOURCES — COMPLETE TECHNICAL DETAILS
--------------------------------------------------------------------------------

SOURCE 1: Google Trends via pytrends
  Install: pip install pytrends
  Usage:
    from pytrends.request import TrendReq
    pytrends = TrendReq(hl='en-US', tz=360)
    pytrends.build_payload(['pottery barn throw blanket'], cat=0, timeframe='today 3-m', geo='US')
    df = pytrends.interest_over_time()
  Output: weekly search index (0-100) for any keyword
  Geography: can specify US states or DMAs
  Rate limits: ~1 request per second to avoid 429 errors; add time.sleep(1)
  Key metric: week-over-week % change in search index = virality detection proxy
  No API key needed. Free. Legal.

SOURCE 2: US Census Building Permits Survey
  URL: https://www.census.gov/construction/bps/
  Download: Monthly data tables as CSV or Excel
  Key fields: metro area, new single-family permits, new multi-family permits, renovation permits
  Update frequency: Monthly (2-week lag)
  API: Also available via Census API at api.census.gov/data/
  Free API key: register at api.census.gov/
  Python: requests library to hit Census API directly
  Key insight: single-family permits → 60-90 day lead for full-home furniture demand
               multi-family permits → 45-60 day lead for smaller-footprint items (West Elm)
               renovation permits → 45-60 day lead for room-specific items

SOURCE 3: FRED Economic Data
  URL: https://fred.stlouisfed.org/
  Free API key: register at fred.stlouisfed.org/docs/api/api_key.html
  Python: pip install fredapi
  Key series:
    - MBAVCH: MBA Purchase Applications Index (weekly, leads home sales by 45 days)
    - MORTGAGE30US: 30-year fixed mortgage rate (weekly)
    - HOUST: Housing starts (monthly)
    - HOUST1F: Single-family housing starts (monthly)
  Usage:
    from fredapi import Fred
    fred = Fred(api_key='YOUR_FREE_KEY')
    mortgage_apps = fred.get_series('MBAVCH', observation_start='2024-01-01')

SOURCE 4: Synthetic sales.csv (provided in PS or generate)
  Format matches PS-provided schema: sku_id, date, units_sold, store_id, category, price
  For hackathon: generate synthetic data with realistic patterns
    - Seasonal baseline (higher in Oct-Dec)
    - Occasional viral spikes (2-5x demand in 1-2 weeks)
    - Slow-moving SKUs with declining trend
  Libraries: numpy, pandas with random seed for reproducibility

SOURCE 5: Inventory.csv (provided in PS or generate)
  Format: sku_id, warehouse_id, stock_on_hand, on_order, lead_time_days
  Key calculation: days_of_supply = stock_on_hand / rolling_avg_daily_sales

SOURCE 6: ACS Census Demographic Data (for brand affinity overlay — optional enhancement)
  URL: census.gov/data/developers/data-sets/acs-5year.html
  Key fields: median household income, median home value (by ZIP or metro)
  Use: map income brackets to WSI brand tier (Pottery Barn vs West Elm vs Williams Sonoma)
  Note: This is a nice-to-have, not required for core MVP

  Rationale: Clean schema integration, no structural changes to existing data model,

Decision 8: Streamlit for dashboard (not React, not Flask+Jinja)
  Rationale: Fastest to build, handles data visualization natively, Python-native
             so ML models can be called directly without API layer

PART 15: CONCERNS, RISKS, AND MITIGATIONS
--------------------------------------------------------------------------------

Risk 1: pytrends rate limiting
  Mitigation: Add time.sleep(1-2) between requests; cache all fetched data to SQLite
              so you never re-fetch during the hackathon; pre-fetch everything on Day 1

Risk 2: FRED API key setup takes time
  Mitigation: Register for key BEFORE the hackathon starts; key is usually instant
              Alternative: hardcode historical FRED data as CSV if key doesn't work

Risk 3: Census permit data is at county/metro level, not ZIP level
  Mitigation: Map metro areas to ZIP code clusters; use geopandas for geographic joins
              This is fine — WSI DCs serve metro clusters, not individual ZIPs

Risk 4: LLM API cost during 56-hour hackathon
  Mitigation: Claude API has generous free tier; cache LLM outputs to avoid repeated calls;
              alternatively use a local Ollama model (Llama 3) — completely free, no API

Risk 5: Synthetic data not convincing enough
  Mitigation: Use real-world inspired patterns with specific numbers (not random noise);
              the viral spike scenario should show realistic magnitude (2-3x baseline);
              use a fixed random seed for reproducibility

Risk 6: 56 hours not enough for all features
  Mitigation: Build order strictly by priority: (1) LLM brief, (2) core forecast,
              (3) dashboard, (4) real estate integration, (5) polish
              If time runs short, cut simulation mode (lowest priority) first

Risk 7: Other teams also pick "Supply Chain Sees the Future"
  Mitigation: The housing permit integration is the moat — no other team will have it
              Also the LLM buyer brief with dollar-impact framing is hard to replicate
              in 56 hours without deliberate planning
